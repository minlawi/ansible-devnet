## Explanation of inv_exercise1.yml

This inventory file defines a hierarchical network device structure in YAML format.

### Structure Overview

```
all:
└── children:
    ├── ce_routers    (8 CE routers: CE1-R9 to CE7-R15, CE6-R16)
    ├── p_routers     (3 P routers: P1-R1, P2-R2, P3-R3)
    ├── pe_routers    (5 PE routers: PE1-R4 to PE5-R8)
    └── routers:
        └── mpls_routers (aggregates all 16 routers above)
```

### What Lines 1-72 Do (Role Groups)

Three router role groups define connection details for each host:
- `ce_routers` - Customer Edge routers (lines 3-36)
- `p_routers` - Provider core routers (lines 37-50)
- `pe_routers` - Provider Edge routers (lines 51-72)

Each host specifies:
- `ansible_connection: ansible.netcommon.network_cli` - Use SSH CLI connection
- `ansible_host: 1.1.1.X` - IP address for management
- `ansible_network_os: cisco.ios.ios` - Cisco IOS platform

### What Lines 73-92 Do (Aggregate Groups)

Creates a `routers` parent group containing `mpls_routers` as a child. The `mpls_routers` group aggregates ALL routers from the role groups into a single flat list using empty dictionaries `{}` as references.

### Why This Structure?

1. **Convenience targeting** - Target all routers with `hosts: mpls_routers` or `hosts: routers`

2. **Avoids repetition** - Empty dicts `{}` reference hosts defined above. Connection info is defined once (lines 5-72), not repeated

3. **Logical organization** - Reflects MPLS topology while maintaining role-based grouping

4. **Playbook flexibility** - Apply common configs to all routers, or target specific roles (CE/P/PE)

### Usage Examples

```yaml
# Target all MPLS routers
- hosts: mpls_routers

# Target only P routers
- hosts: p_routers
```
